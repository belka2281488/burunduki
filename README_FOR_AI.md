# Инструкция для продолжения разработки (для другой нейросети / для себя)

Это браузерный сайт-галерея "Галерея бурундуков" — одностраничное приложение
на чистом HTML/CSS/JS, без сборщиков и фреймворков, с базой данных на Supabase
(Postgres + Storage + Realtime).

## Структура файлов проекта

- `index.html` — вся разметка: галерея, модалки (загрузка фото/видео,
  редактирование профиля, рамки, цвет ника, лайтбоксы и т.д.)
- `style.css` — все стили
- `script.js` — вся логика (один большой файл, разделов много, ищите по
  комментариям и названиям функций)
- `assets/` — картинки: иконки, рамки, курымдык-заглушка и т.д.
- `schema_all.sql` — **справочник** всей схемы базы данных Supabase (см. ниже)

## Как работать с базой данных (важно!)

`schema_all.sql` — это ТОЛЬКО СПРАВОЧНИК для чтения, чтобы видеть полную
картину схемы базы. Все команды в нём уже были применены в Supabase раньше
(в том порядке, в котором идут разделы файла).

**Если нужно добавить в базу что-то новое** (новую таблицу, колонку, функцию,
RLS policy):
1. Создай НОВЫЙ отдельный файл, например `migration_add_<название>.sql`.
2. Пиши туда только новые команды (не копируй туда весь `schema_all.sql`).
3. Этот новый файл нужно будет выполнить в Supabase → SQL Editor.
4. Когда всё заработает и будет протестировано, можно (по желанию)
   попросить объединить содержимое нового файла в конец `schema_all.sql`,
   чтобы файл-справочник снова был актуальным и единственным.

Не редактируй `schema_all.sql` напрямую как "рабочий" файл — он не для этого.

## Технологии и общие принципы проекта

- Supabase используется как: Postgres-база (таблицы + RPC-функции +
  RLS-политики), Storage (для картинок/аватаров/шапок профиля), Realtime
  (для чата активности/уведомлений/онлайн-статусов).
- Идентификация пользователя НЕ через логин/пароль — у каждого браузера
  генерируется случайный `owner_code` (UUID), который хранится в
  `localStorage` под ключом `burunduk-identity`, вместе с именем
  (`{ name, code }`). Это и есть "аккаунт" пользователя. Если человек
  почистит localStorage — он станет "новым" пользователем.
- Внутренняя валюта сайта — "гиперзадки" (таблица `burunduk_gigers`,
  картинка `assets/giperzadka.png`). Ей платят за рамки профиля и за
  курымдык (платный контент).
- Имя пользователя хранится в ДВУХ местах, которые теперь синхронизированы:
  - `localStorage.burunduk-identity.name` — то, что видно в шапке "Ты: ...";
  - `burunduk_profiles.display_name` — то, что видно везде в остальном
    сайте (карточки, лайтбоксы, профиль).
  При смене имени через кнопку "сменить имя" оба места обновляются.
  Везде, где выводится ник автора поста, используется функция
  `displayNameFor(ownerCode, fallbackName)` — она берёт актуальное имя
  из кэша профилей (`profileCache`), а не замороженное `owner_name` из
  самой записи поста (это было источником старого бага "неправильный ник
  на старых постах").

## Основные фичи, которые уже реализованы

1. **Фото и видео галерея** с категориями, поиском, пином постов,
   лайтбоксом с лупой/зумом для фото и встроенным плеером для видео
   (YouTube/Google Drive/прямые ссылки и т.д.)
2. **Рейтинги и комментарии** к постам (с ответами/reply).
3. **"Только для подписчиков"** (`followers_only`) — пост виден только
   подписавшимся на автора либо самому автору.
4. **Профили пользователей** — аватар, шапка (баннер), био, подписчики,
   просмотры, количество публикаций.
5. **Рамки профиля** — покупаются за гиперзадки (wombatramka/burundukramka/
   belkoslavramka) либо разблокируются за количество публикаций
   (patriotramka за 10 постов, capuchinramka за 25 постов). Таблица
   `burunduk_frames`, поле `burunduk_profiles.equipped_frame`.
6. **Цвет/градиент ника** — можно выбрать пресет или задать свой цвет/
   градиент. Хранится в `burunduk_profiles.name_color`. Работает через
   CSS `background-clip: text` для градиентов.
7. **Курымдык** — платный контент за гиперзадки (от 1 до 10). При
   публикации можно включить чекбокс и выбрать цену. Пока не оплачено —
   вместо реального фото/видео/названия показывается заглушка
   `assets/kurymdyk.png` и текст "🎭 Курымдык", автор скрыт. Разблокировка
   постоянная (хранится в БД в `burunduk_kurymdyk_unlocks`, привязана к
   `owner_code` зрителя), переживает перезагрузку страницы.
   **Важно**: логика скрытия курымдыка реализована в ДВУХ разных
   функциях рендера карточек — `renderPhotoGallery`/`renderVideoGallery`
   (основная общая галерея) И `renderProfileGallery` (сетка постов
   именно на странице профиля пользователя). Если будешь трогать
   курымдык — не забудь, что таких мест рендера карточек несколько,
   искать по `isKurymdykLocked(`.
8. **Активность/уведомления** — колокольчик с последними событиями
   (подписки, лайки, комментарии и т.д.), realtime через Supabase.
9. **Онлайн-статус** — зелёная точка у аватара, кто сейчас на сайте
   (Supabase Realtime Presence).

## Как проверить синтаксис после правок

Всегда после правки `script.js` стоит прогнать простой запуск:
```
node -c script.js
```
Это не запускает код, а только проверяет синтаксис на ошибки (незакрытые
скобки, опечатки и т.д.) — быстрый способ отловить явные баги перед
тем, как показывать результат.

## Важные "грабли", уже наступленные в этом проекте (чтобы не повторять)

- Если добавляешь новое поле/чекбокс в модалку загрузки фото/видео —
  не забудь прописать его в ТРЁХ местах: (1) сброс формы при открытии
  "новой" публикации, (2) payload при INSERT, (3) payload при UPDATE
  и подстановка значения при открытии формы РЕДАКТИРОВАНИЯ существующего
  поста. Легко забыть один из трёх и получить баг "работает только при
  создании, а не при редактировании" (или наоборот).
- Если добавляешь что-то, что должно отображаться и в общей галерее,
  и в лайтбоксе, и на странице профиля пользователя — не забывай, что
  это минимум 3-4 разных места рендера карточек в script.js, а не одно.
- `background-clip: text` для градиентного текста требует
  `display: inline-block` и явной установки `-webkit-text-fill-color:
  transparent` — просто класса с `background` недостаточно, CSS-класс
  с явным `color` может перебивать через специфичность, поэтому цвет
  ставится через `el.style.*` (инлайн, выше приоритет), а не только
  через CSS-класс.
- Кэши (`profileCache`, `unlockedKurymdykSet`, `mySubscriptions`) нужно
  аккуратно инвалидировать (`= null` / `delete cache[key]`) при любых
  действиях, которые меняют исходные данные (смена имени, покупка рамки,
  оплата курымдыка, подписка/отписка) — иначе получаются баги вида
  "то показывается, то нет" при разных действиях в разном порядке.
